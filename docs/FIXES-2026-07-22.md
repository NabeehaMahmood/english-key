## 10. Fragile relative asset paths — CSS/JS only loaded at the right URL depth

| | |
|---|---|
| **Files** | `includes/header.php`, `includes/footer.php` |
| **Symptom** | On some pages the stylesheet "didn't load" (page showed unstyled/plain), while `/blog/<slug>` looked fine. |
| **Root cause** | `header.php`/`footer.php` linked assets with **relative** paths (`href="assets/css/style.css"`, `src="assets/js/site.js"`, logo images). A relative path resolves against the *current URL*, so it only works at depth-0 URLs like `/courses.php`. Deep/pretty URLs (e.g. `/blog/<slug>`) would look for `assets/...` under `/blog/`, a 404 — which is exactly why `blog.php`/`blog-post.php` had to inject a `<base href="/">` patch. Every non-blog page was one URL-shape away from losing its CSS, and subfolder deployments were fragile. |
| **Who** | Original template design (predates this branch); surfaced during local testing. |
| **Fix** | Compute a root-absolute `$assetBase` from the **actual request** (`dirname($_SERVER['SCRIPT_NAME'])`) once in `header.php` — NOT from `SITE_URL`, which defaults to `http://localhost/academy` and, when it doesn't match how the site is really served, produced links like `/academy/assets/style.css` that 404 and killed CSS on every page. Prefix the stylesheet, both scripts, and all three logo `<img>`s with it, and append a `?v=<filemtime>` cache-buster. Now assets resolve correctly however the site runs (domain root, XAMPP subfolder, `php -S`) with **zero config**, on every URL depth including `/blog/<slug>`, and the changing `?v=` URL guarantees browsers can't serve a stale/broken `style.css`. Deliberately NOT a global `<base href>`, which would hijack the in-page `#anchor` jump-navs on Home/Courses. |
| **Effect** | Styling/scripts load reliably on all pages regardless of URL shape, install location, or `SITE_URL` config; cache-buster prevents stale-CSS confusion; removes the latent fragility the blog `<base href>` hack was working around. |

## Refactor: `blog.php` / `blog-post.php` `<base href>` now derived from the request, not `SITE_URL`

While reviewing the asset-path fix (#10) for follow-on issues: `blog.php`/`blog-post.php` independently computed their `<base href>` value from `SITE_URL` (`rtrim(parse_url(SITE_URL, PHP_URL_PATH), '/') . '/'`) — the exact same fragile pattern that broke CSS everywhere in #10. Had `SITE_URL` not matched the real deployment, nav links (`href="courses.php"`, still plain relative markup, unlike the now-absolute CSS/JS/logo) on Blog pages would have silently 404'd the same way. Replaced with a reuse of `$assetBase`, the request-derived value `header.php` already computes and exposes in the same PHP scope (`require_once` shares scope with its caller) — one source of truth instead of two, and no `SITE_URL` dependency left in either file. Verified against a live `php -S` instance: `<base href="/">` is now identical and correct across `/blog.php`, `/blog`, and `/blog/<slug>`, and nav links + CSS both resolve correctly on all three.

Also removed a defensive fallback in `includes/footer.php` that recomputed `$assetVer` "in case footer.php is ever rendered without header.php" — traced every one of its ~20 call sites and confirmed every single one runs `includes/header.php` first in the same script, so the fallback was unreachable dead code (violates this codebase's own stated principle: don't add fallbacks for scenarios that can't happen).

## Dev tooling: `router.php` (local-only, mirrors `.htaccess`)

Not a site bug — a gap in local testing. PHP's built-in `php -S` server (used by `run-local.sh`) ignores `.htaccess` entirely, so `/blog` and `/blog/<slug>` never matched their rewrite rules locally. Worse, unmatched paths silently fall through to `index.php` with an HTTP 200, so a broken `/blog/<slug>` request looked "successful" while actually serving the homepage. Added `router.php` (mirrors the two `.htaccess` `RewriteRule`s) and pointed `run-local.sh` at it. Not needed on Apache/XAMPP.
